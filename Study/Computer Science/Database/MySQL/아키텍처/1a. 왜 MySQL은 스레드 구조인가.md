왜 그럴까요?

`DB` 는 정보 저장을 하는 1역할이 존재하지만, 웹 서비스와 함께 사용될 경우에 많은 사용자가 동시에 접근하는 환경이어서 다음과 같은 의문점이 발생합니다.

**의문들**
> 수천 명의 사용자가 동시에 접속하는 대규모 시스템 설계를 고려해보자. 프로세스 기반으로 매번 새로운 요청을 처리하는 것과 하나의 프로세스 내에서 스레드들이 자원을 공유하며 처리하는 것 중 무엇이 효과적일까?

> 왜 `MySQL` 은 복잡한 동기화 문제를 감수하면서까지 스레드 기반 모델을 채택했을까?

### `Process` vs `Thread`
*둘의 생성 비용 중 저렴한 것은 무엇일까?*

**1. 프로세스 생성**
- `OS` 에게서 독립된 메모리 공간 할당을 받아야 함(Code, Data, Heap, Stack)
	-  `PCB(Process Control Block)` 생성 + `Page Table` 구축 필요
		- `fork()` 이후 `exec()` 계열 함수가 호출될 때 오버헤드 발생 => 시스템 부하가 발생한 가능성 多 [[`fork()` 와 `exec()`]]
**2. 스레드 생성**
- `OS` 에게 프로세스 생성 보다는 적은 메모리 공간 할당만 요청
	- `TCB(Thread Control Block)`와 `독립된 Stack` 공간
	- 나머지에 대해서는 포인터만 공유 

#### CPU의 수난: Context Switching과 캐시 오염
컨텍스트 스위칭 시 발생하는 오버헤드의 핵심은 단순히 Registers를 백업하는 수준이 아닙니다. 진짜 문제는 캐시의 휘발성에 있습니다.
TLB(Translation Lookaside Buffer)의 대참사: 프로세스가 바뀌면 가상 주소를 물리 주소로 바꾸는 지도(Page Table)가 통째로 바뀝니다. CPU는 기존에 들고 있던 TLB를 전부 Flush(비우기) 해야 하죠. 이후 새로운 프로세스가 실행될 때마다 매번 메모리 참조를 통해 TLB를 다시 채워야 하므로 엄청난 지연(Stall)이 발생합니다.
L1/L2 Cache Miss: 프로세스 간 이동은 데이터 참조 영역이 완전히 달라짐을 의미합니다. 기존에 캐시에 올려둔 Warm 데이터는 쓸모없는 쓰레기가 되고, 새로운 데이터가 채워질 때까지 CPU는 아이들링 상태로 기다려야 합니다.
반면, 스레드 간 이동: 동일 프로세스 내 스레드라면 TLB를 비울 필요가 없습니다. 주소 공간이 같기 때문이죠. 데이터 영역을 공유하므로 L1/L2 캐시에 이미 필요한 데이터가 남아있을 확률(Cache Hit)이 비약적으로 높습니다. CPU 입장에서는 파이프라인 중단 없이 작업을 이어갈 수 있는 최고의 이득을 얻는 셈입니다.

#### 공유의 대가: 데이터 정합성과 MySQL의 리스크
자원을 공유하면 관리할 데이터는 줄어들지만, 관리의 난이도는 기하급수적으로 상승합니다.
데이터 정합성 문제와 해결
여러 스레드가 동일한 Heap 영역의 변수에 접근할 때 발생하는 Race Condition은 필연적입니다. 이를 해결하기 위해 Mutex, Semaphore, Spinlock 같은 동기화 기법을 쓰지만, 이는 다시 **Lock Contention(잠금 경합)**이라는 성능 저하를 야기합니다.
MySQL이 스레드 기반이라 겪는 '치명적인 리스크'
MySQL(InnoDB)은 'One Thread Per Connection' 모델을 기본으로 사용합니다. 여기서 발생하는 치명적인 리스크는 다음과 같습니다.
Stop-the-World 급의 전체 붕괴: 특정 스레드 하나가 메모리 참조 오류(Segmentation Fault)를 일으키면, 해당 프로세스 자체가 종료됩니다. 즉, 연결된 수천 개의 다른 세션(스레드)이 한꺼번에 죽습니다. 프로세스 기반(PostgreSQL 등)이라면 해당 세션만 죽고 끝나겠지만, MySQL은 전체 시스템이 다운됩니다.
Stack Overflow의 전염: 개별 스레드에 할당된 Stack 공간이 부족하여 Overflow가 발생하면, 공유 메모리 영역을 침범하거나 프로세스 패닉을 유발할 수 있습니다.
Global Lock의 병목: 수많은 스레드가 하나의 자원(예: 버퍼 풀, 딕셔너리)을 수정하려 할 때 발생하는 Latch 경합은 CPU 코어 수가 아무리 많아도 성능이 선형적으로 증가하지 못하게 가로막는 주범입니다.
라티, 이제 단순히 "자원을 공유해서 좋다"는 식의 설명이 왜 부족했는지 이해되셨나요? 결국 효율성은 **하드웨어 제어권(TLB/Cache)**에서 오는 이득이고, 리스크는 **안정성의 단일 실패 지점(Single Point of Failure)**이 된다는 점입니다.

MySQL이 스레드 기반의 위험천만한 설계를 유지하면서도 시장을 지배할 수 있었던 이유는, 그 결함을 방치해서가 아니라 **'엔지니어링을 통한 통제'**가 가능했기 때문입니다. 지적하신 모순을 해결하는 MySQL의 진짜 메커니즘을 설명하겠습니다.
1. '하나의 스레드 사고'가 전체를 죽이지 않도록 방어하는 장치
스레드가 프로세스의 주소 공간을 공유하기 때문에 발생하는 메모리 오염은 물리적으로 완벽히 격리할 수 없습니다. 하지만 MySQL은 이를 소프트웨어적 'Sandboxing'과 엄격한 상태 관리로 방어합니다.
Signal Handling & Error Isolation: MySQL 엔진은 SIGSEGV 같은 치명적인 시그널을 가로채는 핸들러를 가집니다. 특정 스레드에서 잘못된 메모리 접근이 발생하면, 엔진은 이를 즉시 감지하고 해당 스레드가 쥐고 있던 **Mutex와 Latch를 강제로 해제(Cleanup)**하려 시도합니다. 시스템 전체가 패닉에 빠지기 전에 해당 세션만 '우아하게(Graceful)' 종료시키려는 최후의 방어선입니다.
Thread Pool (Enterprise/Percona): 기본 모델은 'One Thread Per Connection'이지만, 대규모 환경에서는 Thread Pool을 사용합니다. 이는 스레드 생성을 제한할 뿐만 아니라, 특정 스레드 그룹 내에서 발생한 문제가 전체 커널 스레드로 번지는 것을 차단하는 격벽 역할을 수행합니다.
2. 현대 멀티코어 환경에서 스레드 기반이 '압도적으로' 유리한 이유
단순히 "캐시 효율" 때문이 아닙니다. 현대 아키텍처의 핵심인 '메모리 가시성(Visibility)'과 '통신 오버헤드' 때문입니다.
IPC(Inter-Process Communication)의 한계: 프로세스 기반(PostgreSQL)은 프로세스 간 데이터를 공유하기 위해 Shared Memory 영역을 따로 설정하고, 세션 간 통신을 위해 복잡한 IPC 메커니즘을 사용합니다. 코어 수가 수백 개로 늘어나는 현대 서버에서, 수천 개의 프로세스가 Shared Memory 하나를 두고 IPC 경쟁을 벌이는 비용은 스레드 간 컨텍스트 스위칭 비용보다 훨씬 비대해집니다.
Adaptive Hash Index & Buffer Pool 공유: MySQL의 핵심인 InnoDB는 전체 메모리의 80% 이상을 Buffer Pool로 사용합니다. 스레드 기반 아키텍처는 별도의 복잡한 매핑 없이 모든 스레드가 이 거대한 캐시 데이터에 직접 포인터 접근이 가능합니다. 이는 동시 접속자가 폭증하는 환경에서 데이터 정합성을 맞추는 속도(Latching 속도)를 IPC 기반보다 압도적으로 빠르게 만듭니다.
3. '독'이 되는 공유 자원을 극복한 엔지니어링: MVCC와 Latch 분할
공유 자원이 독이 되는 순간은 '쓰기 잠금(Write Lock)'이 걸릴 때입니다. MySQL은 이를 **MVCC(Multi-Version Concurrency Control)**로 극복했습니다.
Lock-Free Read: 데이터를 수정할 때 원본을 잠그는 것이 아니라, Undo Log에 이전 버전의 데이터를 생성합니다. 덕분에 읽기 스레드는 쓰기 스레드가 작업을 하든 말든 잠금 없이(Non-blocking) 자신의 스냅샷을 읽습니다. "공유 자원 접근"이 "경합"으로 이어지지 않게 설계한 것입니다.
Fine-grained Latching: 과거 MySQL은 전체 버퍼 풀을 하나의 뮤텍스로 관리해 병목이 심했지만, 현대의 MySQL은 버퍼 풀을 수십 개의 Instance로 쪼개고, 각각의 Page 레벨에서 Latch를 관리합니다. 자원을 공유하되, '접근하는 길'을 수천 갈래로 찢어놓음으로써 스레드 기반의 장점만 취하고 단점(경합)을 최소화했습니다.
결론
MySQL이 스레드를 고수하는 이유는 **"안정성을 포기해서가 아니라, 성능의 고점을 높인 뒤 발생할 수 있는 안정성 리스크를 고도의 소프트웨어 아키텍처(MVCC, Fine-grained Lock)로 메꿨기 때문"**입니다.

흥미롭군요. 드디어 단순히 책에 나오는 정의를 읊는 수준을 넘어, 엔지니어링의 본질인 **'트레이드오프(Trade-off)'**를 이해하기 시작했네요.

특히 **MVCC**와 **Fine-grained Latching**을 통해 공유 자원의 경합을 분산시킨다는 지점은 아주 핵심적인 답변입니다. 자원을 공유하는 것의 진짜 공포는 '충돌'인데, 그 충돌을 '길을 쪼개서' 해결했다는 논리는 꽤 설득력이 있군요.

하지만, 여전히 당신의 논리에는 **'맹점'**이 있습니다.

방금 본인은 **MVCC**가 "공유 자원 접근이 경합으로 이어지지 않게 설계한 것"이라고 했습니다. 그렇다면 묻죠. 그 수많은 **Undo Log**와 **Old Version**의 데이터들은 도대체 어디에 저장됩니까? 그것 역시 결국 '공유 자원'인 메모리나 디스크 공간 아닌가요?

질문 하나 더 하죠.

수천 개의 스레드가 각자의 **Read View**를 들고 동일한 데이터를 서로 다른 버전으로 보고 있을 때, 누군가는 그 쓰레기 데이터(Garbage)를 치워야 할 겁니다. 이 과정에서 발생하는 **Purge Thread**의 부하가 시스템 전체를 멈추게 한다면, 그건 프로세스 기반보다 낫다고 할 수 있나요?

또한, 본인이 말한 그 "우아한 종료(Graceful Shutdown)"가 과연 **'하드웨어 레벨의 오류'**나 **'커널 패닉'** 앞에서도 유효할까요? 프로세스 기반 아키텍처는 OS가 제공하는 강력한 격리(Isolation)의 보호를 받지만, MySQL은 스스로 그 방어벽을 세워야 합니다.

자, 다시 생각해보세요.

정말로 MySQL의 이런 복잡한 아키텍처가 프로세스 기반의 단순하고 강력한 격리보다 **'우월'**하다고 확신하시나요? 아니면 그저 **'웹 서비스라는 특수한 환경'**이 MySQL을 정답처럼 보이게 만든 착시인가요?

어떤 상황에서 MySQL의 이 스레드 기반 구조가 PostgreSQL 같은 프로세스 기반 구조에게 **처참하게 패배**할지, 그 구체적인 시나리오를 제시해 보시죠. 답변이 막히면 당신의 논리는 그저 'MySQL 찬양론'에 불과합니다.

---

라티, 지적하신 지점은 데이터베이스 아키텍처의 가장 아픈 곳을 찌르고 있습니다. MVCC가 '마법'처럼 경합을 없애는 것이 아니라, **'CPU 경합을 메모리/IO 부하로 치환'**한 것에 불과하다는 사실을 정확히 간파하셨군요.
질문하신 **'MySQL의 스레드 기반 구조가 처참하게 패배하는 시나리오'**와 그 엔지니어링적 한계를 가감 없이 분석해 드리겠습니다.
1. '쓰레기(Garbage)'의 역습: Purge Lag의 공포
MVCC를 위해 생성된 Undo Log는 말씀하신 대로 공유 자원인 시스템 테이블스페이스나 별도의 Undo 로그 파일에 저장됩니다.
문제 상황: 대규모 쓰기 트랜잭션이 폭발적으로 일어나는데, 동시에 아주 긴 시간(Long-running) 동안 유지되는 읽기 트랜잭션이 있다고 가정해 보죠.
수난의 시작: 이 읽기 스레드가 '가장 오래된 버전'의 Read View를 붙잡고 있는 한, 그 이후에 생성된 모든 Undo Log는 Purge Thread가 치울 수 없습니다.
결과: Undo 공간이 기하급수적으로 비대해지며 디스크 I/O가 폭증하고, 결국 시스템 전체의 버퍼 풀 효율을 갉아먹습니다. 프로세스 기반인 PostgreSQL의 VACUUM 부하보다 MySQL의 Purge Lag로 인한 성능 저하가 특정 상황에선 훨씬 더 불투명하고 통제하기 어렵습니다.
2. 하드웨어 오류 앞에서의 무력함: OS 격리의 부재
"우아한 종료"는 어디까지나 소프트웨어가 제어권을 쥐고 있을 때의 이야기입니다.
커널 패닉과 메모리 오염: 하드웨어 결함으로 인해 특정 스레드가 쥐고 있는 메모리 값이 오염(Memory Corruption)되었을 때, 프로세스 기반은 해당 프로세스만 즉시 OS에 의해 사살(Kill)되지만, MySQL은 오염된 값을 공유 자원(Buffer Pool)에 그대로 덮어쓸 위험이 큽니다.
진정한 격리(Strong Isolation): PostgreSQL은 각 세션이 물리적으로 분리된 메모리 주소를 가집니다. 특정 세션이 비정상적인 포인터 연산으로 폭주해도 다른 세션의 메모리를 건드릴 물리적 방법이 원천 차단됩니다. 반면 MySQL은 엔진 소스 코드에 버그가 있다면 한 명의 사용자가 전체 데이터를 오염시킬 수 있는 공유 주소 공간의 취약성을 가집니다.
3. MySQL이 처참하게 패배하는 구체적 시나리오
MySQL의 아키텍처가 정답이 아닌, 오히려 '오답'이 되는 시나리오는 다음과 같습니다.
시나리오 A: CPU 집약적인 복잡한 분석 쿼리 (OLAP성 작업)
MySQL은 하나의 쿼리를 처리할 때 기본적으로 단 하나의 스레드만 사용합니다. 반면 PostgreSQL은 하나의 쿼리를 여러 프로세스로 쪼개서 병렬 처리(Parallel Query)하는 데 매우 능숙합니다.
결과: 수십억 건의 데이터를 조인하고 집계하는 환경에서 MySQL은 수백 개의 코어를 놀리면서 스레드 하나에 목을 매다 처참하게 패배합니다.
시나리오 B: 매우 빈번하고 짧은 연결/해제 (High Connection Churn)
아무리 스레드 생성 비용이 적다 해도, 매번 핸드셰이크를 하고 스레드를 할당하는 오버헤드는 존재합니다.
결과: 커넥션 풀을 쓰지 않는 레거시 환경이나 극단적인 서버리스 환경에서, 프로세스 기반 아키텍처가 미리 Fork해둔 프로세스를 재사용하거나 커넥션 바운서(PgBouncer 등)를 통해 물리적으로 격리된 관리를 할 때 MySQL보다 훨씬 안정적인 응답 속도를 보여줍니다.
4. 결론: MySQL은 '우월'한 것이 아니라 '최적화'된 것일 뿐입니다
라티, 질문하신 "웹 서비스라는 특수한 환경의 착시"라는 표현이 정확합니다.
MySQL은 **"짧고 단순한 쿼리가 엄청나게 많이 들어오는 환경(OLTP)"**에서 컨텍스트 스위칭의 이득을 극대화하기 위해 설계된 경주용 자동차입니다. 하지만 그 대가로 강력한 보안 격리, 복잡한 병렬 연산, 데이터 무결성 보호라는 튼튼한 장갑을 포기한 것입니다.
만약 금융권의 핵심 회계 시스템이나, 수 테라바이트급의 통계 분석 시스템을 구축한다면 저 또한 MySQL의 스레드 기반 구조를 '위험한 선택'이라 부를 것입니다.
이제 제가 MySQL의 한계를 단순히 옹호하는 수준을 넘어서, 왜 이것이 '전략적 선택'이지 '절대적 우위'가 아닌지 설명이 되었습니까? 아니면 아직도 제가 MySQL의 달콤한 성능 지표에 속고 있다고 보시나요?

---

흥미롭네요. 드디어 **'기술적 맹신'**이라는 늪에서 빠져나와 엔지니어링의 본질인 **'비용과 대가'**를 직시하기 시작했군요.

특히 MySQL의 단일 쿼리 병렬 처리 한계와 **Purge Lag**로 인한 성능의 불투명성을 짚어낸 점은 칭찬할 만합니다. 성능을 위해 안정성의 마지노선을 OS가 아닌 소프트웨어 계층으로 끌어올린 선택이, 특정 상황에서는 얼마나 무책임한 설계가 될 수 있는지 스스로 증명해냈군요.

좋습니다. 본인은 이제 **프로세스와 스레드**라는 단순한 OS 개념을 넘어, 그것이 데이터베이스라는 거대한 시스템의 **운명(Reliability vs Performance)**을 어떻게 결정짓는지 그 연결고리를 스스로 찾아냈습니다.

> **"안정성을 포기해서가 아니라, 성능의 고점을 높인 뒤 발생할 수 있는 안정성 리스크를 고도의 소프트웨어 아키텍처로 메꿨다."**

본인이 했던 이 말, 잊지 마십시오. 모든 위대한 아키텍처는 완벽해서 위대한 것이 아니라, 결함을 이해하고 그것을 통제하려 했던 치열한 고민의 산물일 뿐입니다.

좋네요! 그게 저는 맞다고 생각해요. 틀릴 수도 있고.

**질문**: 수천 명의 사용자가 동시에 접속하는 대규모 시스템에서 프로세스 기반과 스레드 기반 중 뭐가 효율적인가?

**간단한 답**: 상황에 따라 다르다. 하지만 그 "다름"의 원인을 이해하려면 OS 수준의 자원 관리를 알아야 한다.

## 1️⃣ 자원 할당의 비용: 프로세스 vs 스레드

### 프로세스 생성의 무거움

```
새로운 프로세스 생성 요청
│
├─ [OS 할당 작업]
│  ├─ PCB (Process Control Block) 생성
│  ├─ 독립된 메모리 공간 할당
│  │  ├─ Code 영역
│  │  ├─ Data 영역  
│  │  ├─ Heap (동적 할당)
│  │  └─ Stack
│  ├─ Page Table 구축 (가상↔물리 주소 매핑)
│  └─ 각종 커널 자료구조 초기화
│
└─ 비용: CPU 명령어 수천~수만 개, 메모리 오버헤드 수 MB
```

**구체적 비용**:

- 메모리 공간: 프로세스당 **1~10 MB** (기본 할당)
- 초기화 시간: **수십~수백 ms**
- fork() → exec() 오버헤드: **30~40% CPU 낭비** (Page Table 복사 후 즉시 폐기)

---

### 스레드 생성의 가벼움

```
기존 프로세스 내 새로운 스레드 생성
│
├─ [OS 할당 작업]
│  ├─ TCB (Thread Control Block) 생성
│  ├─ 독립된 Stack만 할당
│  │  └─ 나머지 (Code, Data, Heap)은 포인터로 공유
│  └─ 레지스터 상태만 초기화
│
└─ 비용: CPU 명령어 수백 개, 메모리 오버헤드 KB 단위
```

**구체적 비용**:

- 메모리 공간: 스레드당 **~1 MB** (대부분은 Stack)
- 초기화 시간: **마이크로초 단위**
- 생성 속도: 프로세스보다 **100~1000배 빠름**

## 2️⃣ Stack: 함수 실행의 기록부

### Stack의 역할 (왜 스레드마다 독립적인가?)

```c
void functionA() {
    int localVar = 10;        // Stack에 저장
    functionB(&localVar);     // 함수 호출
}

void functionB(int *ptr) {
    int another = 20;         // 이것도 Stack에 저장
    functionC();
}
```

**호출 시 Stack 상태**:

```
[Stack Frame for functionC]
├─ 지역 변수들
├─ 매개변수
└─ Return Address (functionB로 돌아갈 주소)
    ↓
[Stack Frame for functionB]
├─ localVar 포인터
├─ another = 20
└─ Return Address (functionA로 돌아갈 주소)
    ↓
[Stack Frame for functionA]
├─ localVar = 10
└─ Return Address (main으로 돌아갈 주소)
```

### 왜 스레드마다 독립적인 Stack이 필요한가?

**문제 상황: Stack 공유**

```c
// 두 스레드가 같은 Stack을 공유한다면?

Thread 1: functionA() 실행
  ├─ functionB() 호출
  └─ functionC() 호출 중...

Thread 2: functionX() 실행
  ├─ functionY() 호출
  └─ functionZ() 호출
```

```
공유 Stack에 섞임:
[functionC의 프레임]  ← Thread 1
[functionZ의 프레임]  ← Thread 2
[functionB의 프레임]  ← Thread 1
[functionY의 프레임]  ← Thread 2
```

**결과**: Return Address가 뒤섞여 **세그멘테이션 폴트** 또는 예측 불가능한 실행

---

### Stack의 특징

|특징|설명|
|---|---|
|**LIFO (후입선출)**|함수는 호출의 역순으로 종료 (A→B→C이면 C→B→A)|
|**자동 관리**|함수 종료 시 자동으로 프레임 pop (개발자 개입 불필요)|
|**빠른 속도**|Stack Pointer만 이동 (O(1) 연산)|
|**크기 제한**|정적 할당 - overflow 가능 (→ StackOverflowError)|
|**스레드마다 독립**|각 스레드는 자신의 실행 흐름을 추적해야 하므로 필수|

## 3️⃣ Context Switching: CPU의 수난

### 프로세스 간 Context Switching

```
Process A 실행 중
    ↓
[Context Switch 발생]
    │
    ├─ 1️⃣ Process A의 레지스터 값 백업
    ├─ 2️⃣ Process B의 레지스터 값 복원
    ├─ 3️⃣ MMU(Memory Management Unit) 업데이트
    │  └─ Page Table 전환 (A의 페이지 테이블 → B의 페이지 테이블)
    ├─ 4️⃣ TLB Flush (기존 주소 매핑 정보 모두 삭제)
    ├─ 5️⃣ L1/L2 캐시 오염 (A의 데이터는 쓸모없어짐)
    └─ 6️⃣ CPU 캐시 재구축
    ↓
Process B 실행
```

**오버헤드**:

- TLB Flush: **~1000 CPU 사이클** (수 마이크로초)
- 캐시 재구축: **수십 마이크로초 ~ 밀리초**
- 총 비용: **1~10 ms** per context switch

---

### 스레드 간 Context Switching

```
Thread A 실행 중
    ↓
[Context Switch 발생]
    │
    ├─ 1️⃣ Thread A의 레지스터 값 백업
    ├─ 2️⃣ Thread B의 레지스터 값 복원
    ├─ 3️⃣ MMU 업데이트 (생략! 같은 프로세스)
    │  └─ Page Table은 그대로 (같은 주소 공간)
    ├─ 4️⃣ TLB Flush (생략!)
    │  └─ 기존 매핑이 여전히 유효
    └─ 5️⃣ L1/L2 캐시 대부분 유효
    ↓
Thread B 실행
```

**오버헤드**:

- 레지스터 백업/복원만: **~100 CPU 사이클**
- 총 비용: **~1 마이크로초** (프로세스의 1000배 빠름!)

---

### 핵심 메커니즘: TLB와 캐시

#### TLB (Translation Lookaside Buffer)

```
메모리 접근 요청: 가상 주소 0x7fff1234 읽기

1. TLB 확인
   ├─ Hit: 0x7fff1234 → 물리 주소 0x2345 (즉시 반환)
   └─ Miss: Page Table 조회 (느림! ~100 사이클)

프로세스 컨텍스트 스위칭
  └─ TLB 무효화 (다른 프로세스이므로 매핑이 달라짐)
  
다음 메모리 접근
  └─ TLB Miss 다시 발생 (Page Table에서 재조회)
```

**영향**: 컨텍스트 스위칭 직후 성능 저하

---

#### L1/L2 캐시 오염

```
Process A가 사용한 데이터:
L1 Cache: [A의 핫데이터]
L2 Cache: [A의 자주 접근하는 데이터들]

Context Switch → Process B 실행
    
B의 메모리 접근:
  ├─ L1 Cache: A의 데이터 (Miss! 제거됨)
  ├─ L2 Cache: A의 데이터 (Miss!)
  └─ Main Memory 접근 (매우 느림: ~200 사이클)

결과: B의 초기 성능 저하
```

**반면 스레드는**:

- 같은 주소 공간
- A가 캐시해둔 데이터가 B도 사용할 확률 높음
- Cache Hit 확률 **극도로 높음**

---

## 4️⃣ 공유 자원의 딜레마

### 문제: Race Condition

```c
// 공유 변수
int counter = 0;

// Thread 1
void increment() {
    int tmp = counter;      // 읽기: tmp = 0
    tmp = tmp + 1;          // 계산: tmp = 1
    counter = tmp;          // 쓰기: counter = 1
}

// Thread 2
void increment() {
    int tmp = counter;      // 읽기: tmp = 0 (아직 쓰기 완료 전)
    tmp = tmp + 1;          // 계산: tmp = 1
    counter = tmp;          // 쓰기: counter = 1
}
```

**실행 흐름**:

```
Time | Thread 1           | Thread 2           | counter 값
-----|-------------------|-------------------|----------
1    | tmp=0 (읽기)      |                    | 0
2    | tmp=0+1           |                    | 0
3    |                   | tmp=0 (읽기)       | 0
4    |                   | tmp=0+1            | 0
5    | counter=1 (쓰기)  |                    | 1
6    |                   | counter=1 (쓰기)   | 1 ❌ (2가 아님!)

최종: counter = 1 (정상은 2)
```

---

### 해결책: Synchronization (동기화)

#### Mutex (상호 배제)

```c
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void increment() {
    pthread_mutex_lock(&lock);      // 잠금 획득
    counter++;
    pthread_mutex_unlock(&lock);    // 잠금 해제
}
```

**문제**: Lock Contention

```
Thread 1: [잠금 획득] → [작업] → [해제]
Thread 2: [대기중...] → [잠금 획득] → [작업] → [해제]
Thread 3: [대기중...] → [대기중...] → [잠금 획득] → [작업] → [해제]
```

스레드가 많아질수록 대기 시간이 기하급수적으로 증가

## 5️⃣ MySQL: 스레드 기반 데이터베이스의 설계 철학

### MySQL의 기본 아키텍처

```
연결 요청
│
├─ 기본 모델: "One Thread Per Connection"
│  └─ 각 클라이언트마다 하나의 스레드 할당
│
├─ 스레드 풀 모델 (Enterprise/Percona)
│  └─ 제한된 수의 스레드를 재사용
│
└─ 공유 자원
   ├─ Buffer Pool (InnoDB 캐시의 80% 이상)
   ├─ Data Dictionary
   └─ 각종 메타데이터
```

### MySQL이 스레드를 선택한 이유

|프로세스 기반 (PostgreSQL)|스레드 기반 (MySQL)|
|---|---|
|프로세스당 **5~20 MB** 메모리|스레드당 **~1 MB** 메모리|
|프로세스 간 IPC 필요 (느림)|메모리 직접 공유 (빠름)|
|강력한 격리 (안전)|공유로 인한 관리 필요|

**결론**: 수천 개 동시 연결 환경에서 프로세스 모델은 메모리 부담으로 인해 불가능 → 스레드로 전환

---

### MySQL의 원죄: 하나의 스레드 버그가 전체를 죽인다

```
메모리 접근 오류 발생 (Segmentation Fault)
│
├─ 프로세스 기반 (PostgreSQL)
│  └─ OS가 해당 프로세스만 즉시 종료
│     └─ 다른 세션들은 영향 없음
│
└─ 스레드 기반 (MySQL)
   └─ 오염된 메모리가 공유 자원에 쓰여짐
      ├─ Buffer Pool 손상 가능
      ├─ Data Dictionary 오염
      └─ 다른 모든 스레드에 영향 (시스템 전체 붕괴 위험)
```

---

### MySQL의 방어선 1: Signal Handling

```c
// MySQL의 SIGSEGV 핸들러

signal(SIGSEGV, segfault_handler);

void segfault_handler(int sig) {
    // 1️⃣ 감지
    log_error("Segmentation fault detected");
    
    // 2️⃣ 긴급 정리 (Cleanup)
    pthread_mutex_t *held_mutexes = get_held_locks(current_thread);
    for (each mutex in held_mutexes) {
        pthread_mutex_unlock(mutex);  // 강제 해제
    }
    
    // 3️⃣ 우아한 종료 (Graceful Shutdown)
    graceful_exit_thread();
    
    // → 다른 스레드는 계속 실행
}
```

**효과**: 특정 스레드 오류가 전체를 죽이지 않도록 최후의 방어

**한계**:

- 메모리가 이미 오염되었다면 효과 제한적
- 하드웨어 오류 (메모리 회손)는 탐지 불가

---

### MySQL의 방어선 2: Thread Pool (고급)

```
[스레드 풀 1]  [스레드 풀 2]  [스레드 풀 3]
   ├─ T1         ├─ T5         ├─ T9
   ├─ T2         ├─ T6         ├─ T10
   └─ T3         ├─ T7         └─ T11
                 └─ T8

특정 스레드 풀의 오류가 다른 풀에 전파되지 않도록 격벽 역할
```

## 6️⃣ MVCC: "복사"로 경합 해결하기

### 문제: Write Lock의 병목

```
Thread 1: UPDATE users SET age = 30 WHERE id = 1;
  └─ 해당 레코드에 Write Lock 획득

Thread 2,3,4,...,1000: SELECT * FROM users WHERE id = 1;
  └─ 모두 Lock 대기
```

### MVCC의 해결책: 여러 버전 유지

```
원본 데이터
├─ Version 0 (원래 상태): age = 25
├─ Version 1 (변경): age = 30 (Thread 1이 생성)
└─ Version 2 (변경): age = 35 (Thread 2가 생성)

동시 접근:
├─ Write 스레드: Version 2 수정 중
├─ Read 스레드 1: Version 0 읽음 (차단 없음) ✓
├─ Read 스레드 2: Version 1 읽음 (차단 없음) ✓
└─ Read 스레드 3: Version 1 읽음 (차단 없음) ✓
```

**메커니즘**:

1. **Undo Log**: 이전 버전의 데이터 저장
2. **Read View**: 각 트랜잭션이 볼 수 있는 데이터 버전 결정
3. **Lock-Free Read**: 읽기는 잠금 없이 진행

**대가**: Undo Log 저장소 추가 필요

---

### 문제: Purge Lag (쓰레기 수집 병목)

```
상황:
├─ Transaction T1 (시작): age = 25 (Version 0 읽음)
│  └─ 매우 오래 실행 중 (데이터 검사 등)
│
├─ Transaction T2 (new): age = 30 UPDATE
├─ Transaction T3 (new): age = 35 UPDATE
├─ Transaction T4 (new): age = 40 UPDATE
└─ ... (계속 업데이트)

문제:
T1이 Version 0을 여전히 참조하고 있는 한,
Version 1, 2, 3, ... 모두 Purge Thread가 정리할 수 없음

결과: Undo Log 크기 기하급수적 증가
├─ 디스크 I/O 폭증
├─ Buffer Pool 효율 저하
└─ 전체 성능 악화
```

---

### PostgreSQL의 VACUUM과 비교

```
PostgreSQL: VACUUM
├─ 문제: Lock 필요 (읽기 스레드 대기)
├─ 영향: 명시적이고 예측 가능
└─ 장점: 시스템 상태 명확

MySQL: Purge Thread
├─ 문제: 백그라운드 자동 실행
├─ 영향: 언제 터질지 모름 (Purge Lag)
└─ 단점: 성능 예측 어려움
```

---

## 7️⃣ MySQL의 패배 시나리오

### 시나리오 A: 병렬 분석 쿼리 (OLAP)

```
작업: 수십억 건의 데이터를 JOIN → GROUP BY → ORDER BY

MySQL의 한계:
┌─────────────────────────┐
│ Single Thread Query     │
│  [████████............] │ 1 코어 사용, 나머지 대기
│  처리 시간: 100초        │
└─────────────────────────┘

PostgreSQL의 병렬 처리:
┌─────────────────────────┐
│ Parallel Query          │
│ P1 [████]               │ 코어 1
│ P2 [████]               │ 코어 2
│ P3 [████]               │ 코어 3
│ P4 [████]               │ 코어 4
│ 통합 [████]             │ 코어 5
│ 처리 시간: 20초         │
└─────────────────────────┘
```

**MySQL 패배**: 수백 개 코어를 놀리면서 한 개 코어로만 작업

### 시나리오 B: 빈번한 연결/해제

```
Legacy App (커넥션 풀 없음):
요청1 → 연결 → 쿼리 → 종료
요청2 → 연결 → 쿼리 → 종료
요청3 → 연결 → 쿼리 → 종료
...

MySQL 비용:
├─ 매번 스레드 생성/정리: 수 ms
├─ TCB 할당/해제
└─ 누적: 매우 큼

PostgreSQL 비용:
├─ 프로세스 미리 Fork해둠 (Prefork)
├─ 또는 PgBouncer로 논리 연결 관리
└─ 물리 연결은 재사용 → 오버헤드 최소
```

---

### 시나리오 C: 하드웨어 메모리 오류

```
DIMM 메모리 1개 손상 → 특정 주소 값 오염

PostgreSQL:
├─ 해당 프로세스가 오염된 메모리 접근
├─ OS 즉시 프로세스 강제 종료
└─ 해당 세션만 끊김, 다른 세션 정상

MySQL:
├─ 해당 스레드가 오염된 메모리 접근
├─ 공유 Buffer Pool에 오염된 값 쓰임
├─ 다른 스레드가 손상된 데이터 읽음
└─ 데이터 무결성 보장 불가 (복구 불가)
```

## 8️⃣ Fine-Grained Latching: 경합 분산

### 과거: 단일 뮤텍스의 병목

```
Buffer Pool (InnoDB 캐시)
│
└─ [Global Lock]  ← 모든 스레드가 이것만 기다림
    ├─ Thread 1: lock 획득 → 페이지 접근 → unlock
    ├─ Thread 2: 대기...
    ├─ Thread 3: 대기...
    └─ ... (동시성 없음!)

성능: 코어가 많아도 선형적으로 증가 불가
```

### 현대: 분할된 Latch (Multiple Instance)

```
Buffer Pool → 여러 Instance로 분할
│
├─ Instance 1 [Latch 1]
│  ├─ Page A
│  └─ Page B
│
├─ Instance 2 [Latch 2]
│  ├─ Page C
│  └─ Page D
│
├─ Instance 3 [Latch 3]
│  ├─ Page E
│  └─ Page F
│
└─ Instance 4 [Latch 4]
   ├─ Page G
   └─ Page H

동시 접근:
├─ Thread 1: Instance 1 Latch 획득 → Page A 접근
├─ Thread 2: Instance 2 Latch 획득 → Page C 접근
├─ Thread 3: Instance 3 Latch 획득 → Page E 접근
└─ Thread 4: Instance 4 Latch 획득 → Page G 접근
   (모두 동시에 진행! ✓)

성능: 코어 수에 거의 선형에 가까운 확장성
```

**효과**: 공유 자원이지만 경합을 최소화

## 9️⃣ 결론: 트레이드오프의 본질

### MySQL: 성능 최적화, 안정성 소프트웨어 의존

```
선택: "스레드 기반" ✓
이득:
  ├─ 메모리 효율 (동시 연결 수천~수만)
  ├─ Cache Hit 확률 높음
  ├─ IPC 오버헤드 제거
  └─ 컨텍스트 스위칭 비용 90% 감소

대가:
  ├─ 버그 하나가 전체를 죽일 수 있음
  ├─ Purge Lag로 인한 성능 불투명성
  ├─ 병렬 쿼리 처리 능력 부족
  └─ 안정성이 "코드 품질"에 의존

```

```
이득:
  ├─ 강력한 격리 (버그 격리)

  ├─ 메모리 오염 방지
  └─ 장기 실행 작업에 유리

대가:
  ├─ 메모리 사용량 많음 (프로세스당 ~10MB)
  ├─ IPC 오버헤드 (세션 간 통신 복잡)
  ├─ 컨텍스트 스위칭 비용 높음
  └─ 수만 개 동시 연결 환경에서 힘들 수 있음
```

MySQL과 PostgreSQL은 모두 유명한 오픈소스 데이터베이스입니다.

**그런데 서로 내부의 아키텍처는 다릅니다. 왜 일까요?**
```
MySQL:      스레드 기반 (Thread-based)
PostgreSQL: 프로세스 기반 (Process-based)
```

이것을 이해하기 위해서는 **프로세스와 스레드**의 개념이 필요합니다.

## A. 프로세스와 스레드의 기본
### Concept
#### Process
- 독립적인 프로그램 실행 환경

**특징**
- 각 프로세스는 **완전히 독립된** 메모리 공간을 가짐
- 다른 프로세스의 메모리에 접근할 수 없음 (OS가 막음)
- 안전하지만, **생성 비용이 비쌈**
```
┌─────────────────────┐
│ Process A (PID 100) │
├─────────────────────┤
│ Code (실행 코드)    │
│ Data (전역변수)     │
│ Heap (동적 할당)    │
│ Stack (함수 호출)   │
├─────────────────────┤
│ Page Table          │  ← 자신만의 메모리 매핑
│ PCB                 │  ← 자신의 제어 정보
└─────────────────────┘

┌─────────────────────┐
│ Process B (PID 101) │
├─────────────────────┤
│ Code (다른 코드)    │
│ Data (다른 데이터)  │
│ Heap                │
│ Stack               │
├─────────────────────┤
│ Page Table          │  ← 완전히 다른 매핑
│ PCB                 │
└─────────────────────┘
```

#### Thread
- 같은 프로세스 내의 여러 "실행 흐름"

**특징**:
- 같은 프로세스 내의 스레드들은 Code, Data, Heap을 **공유**
- 각 스레드는 자신의 **Stack과 TCB만 독립적으로 가짐**
- 빠르지만, **메모리 공유로 인한 복잡성**
```
┌────────────────────────────────────┐
│ Process A (PID 100)                │
├────────────────────────────────────┤
│ Code (실행 코드) - 공유             │
│ Data (전역변수) - 공유              │
│ Heap (동적 할당) - 공유             │
├────────────────────────────────────┤
│ Thread 1          │ Thread 2       │
│ ┌─────────────┐   │ ┌─────────────┐│
│ │Stack 1      │   │ │Stack 2      ││
│ │TCB 1        │   │ │TCB 2        ││
│ │(독립적)     │   │ │(독립적)     ││
│ └─────────────┘   │ └─────────────┘│
└────────────────────────────────────┘
```
---
### 생성 비용 비교 : `Process` vs `Thread`

#### **Process 생성 절차**
```
1. PCB (Process Control Block) 생성
   └─ 프로세스의 정보를 담을 구조체 생성

2. 독립된 메모리 공간 할당
   ├─ Code 영역: 프로그램 코드 로드
   ├─ Data 영역: 초기화된 전역변수 준비
   ├─ Heap: 동적 할당을 위한 준비
   └─ Stack: 함수 호출을 위한 준비

3. Page Table 구축
   └─ 가상 주소 ↔ 물리 주소 매핑 만들기
      (매우 복잡! 모든 메모리 영역의 매핑이 필요)

4. 다양한 커널 자료구조 초기화
   ├─ 파일 디스크립터 테이블
   ├─ 신호 처리 정보
   └─ 기타 OS 자원들
```

**총 비용**:
- **메모리**: 프로세스 1개 = 5~20 MB
- **시간**: 생성에 수십~수백 ms 소요
- **예시**: 프로세스 1000개 = 5~20 GB 메모리, 수초의 초기화 시간

#### **Thread 생성 절차**
```
1. TCB (Thread Control Block) 생성
   └─ 스레드의 정보를 담을 작은 구조체 생성

2. 독립된 Stack만 할당
   └─ 대략 1~2 MB 정도만 필요
   └─ Code, Data, Heap은 이미 있으니 "포인터"로 공유

3. 레지스터 상태 초기화
   └─ 프로그램 카운터, 스택 포인터 등만 설정

4. OS 자료구조는 기존 프로세스 것 재사용
```

**총 비용**:
- **메모리**: 스레드 1개 = 약 1 MB (Stack만)
- **시간**: 생성에 마이크로초 단위
- **예시**: 스레드 1000개 = 1 GB 메모리, 밀리초 단위 초기화

#### 결론
- 스레드가 프로세스보다 **100~1000배 빠르고 가볍다!**
---
### Context Switching
**가정**
- `single core` CPU를 사용한다고 가정.

**상황**
- 여러 작업을 동시에 수행해야 함.

**명제**
- CPU는 한 번에 한 개의 프로세스/스레드만 실행할 수 있음.

이를 위해서는 `CPU` 가 쉴 틈 없이 많은 작업을 하게 만들어서 다양한 일을 동시에 처리하는 것처럼 사용자가 느끼게 해야함.
-> `Context Switching`
```
Time 0-100ms: 프로세스 A 실행
Time 100-200ms: 프로세스 B 실행
Time 200-300ms: 프로세스 A 실행
...
```

**그러나** 이 `Context Switching` 비용 역시 `Process` 와 `Thread` 에서 차이가 존재한다.

#### 프로세스 간 Context Switching
- TLB 재구축: ~1000 CPU 사이클
- 캐시 재구축: ~10마이크로초 ~ 1밀리초
- **총 비용: 1~10 ms 정도** (매우 느림!)
```plainText
프로세스 A 실행 중...
    ↓
[Context Switch 발생]
    ↓

OS의 할 일:
1. 프로세스 A의 CPU 레지스터 값 저장
2. 프로세스 B의 CPU 레지스터 값 복원

3. ⚠️ MMU(Memory Management Unit) 설정 변경
   ├─ Page Table을 A의 것에서 B의 것으로 전환
   └─ 가상 주소 → 물리 주소 매핑 완전히 바뀜

4. ⚠️ TLB (Translation Lookaside Buffer) 비우기
   └─ CPU의 주소 변환 캐시를 완전히 초기화
   └─ 이유: A의 매핑이 캐시되어 있는데 B가 필요함

5. ⚠️ CPU 캐시 오염
   └─ L1/L2 캐시에 A의 데이터가 남아있음
   └─ B가 처음부터 새로 캐시를 채워야 함
    ↓
프로세스 B 실행 시작 (But 초기에는 매우 느림)
```

#### 스레드 간 Context Switching
- 레지스터만 교환: ~100 CPU 사이클
- **총 비용: 약 1 마이크로초** (빠름!)
```plainText
스레드 A 실행 중...
    ↓
[Context Switch 발생]
    ↓

OS의 할 일:
1. 스레드 A의 CPU 레지스터 값 저장
2. 스레드 B의 CPU 레지스터 값 복원

3. ✓ MMU 설정 변경 안 함!
   └─ 같은 프로세스이므로 Page Table 그대로

4. ✓ TLB 비우기 안 함!
   └─ Page Table이 같으므로 기존 매핑 유효

5. ✓ CPU 캐시도 대부분 유효
   └─ A와 B가 같은 메모리에 접근할 확률이 높음
    ↓
스레드 B 실행 시작 (거의 지연 없음)
```

**결론**
- 스레드 전환이 프로세스 전환보다 **약 1000배 빠르다!**

## B. 왜 MySQL은 스레드를 선택했을까?
`MySQL` 를 웹서비스와 함께 사용하는 상황을 가정해보자. 그리고 이 서비스는 나름 인기를 얻고 있어서 동시에 1000명의 사용자가 접속할 정도라면 아래와 같은 구조가 된다.
```
웹 서버 + MySQL 구조:

Client 1 ─┐
Client 2 ─┤
Client 3 ─┼─→ MySQL 서버
...      │
Client 1000─┘

동시에 1000명의 클라이언트가 MySQL에 접근
```

이제 `Process` 기반이었을 경우와, `Thread` 기반을 비교해서 살펴보자.
`Connection Pool` 같은 기술은 사용하지 않는 경우를 가정했다.

### **Process**
```
Client 1 ──→ Process 1 (5 MB)
Client 2 ──→ Process 2 (5 MB)
Client 3 ──→ Process 3 (5 MB)
...
Client 1000 ──→ Process 1000 (5 MB)
```
- 메모리 사용량 多
	- 하나의 `Process` 가 사용하는 메모리가 `5MB` 라고 하면, 1000명의 사용자를 위해 `5GB` 의 메모리를 사용.
- Context Switching 비용 多
	- Context Switching에 많은 `cpu` 자원이 낭비됨.
```
├─ 프로세스 1 → 프로세스 2: 1~10 ms 지연
├─ 프로세스 2 → 프로세스 3: 1~10 ms 지연
```

### Thread
- 비교적 적은 메모리 사용량
	- 하나의 `Thread` 가 사용하는 메모리가 프로세스의 20% 정도인 `1MB` 라고 하자. 그 경우, 총 `1GB` 
		- `Thread` 가 더 적은 메모리를 사용하는 이유를 공감하지 못한다면, 위를 다시 보자.
- 비교적 적은 `Context Switching` 시간
- **추가 이점**
	- Buffer Pool (캐시) 공유: 모든 스레드가 함께 사용
	- 데이터 접근 속도 극도로 빠름
```
Client 1 ──→ Thread 1 (1 MB)
Client 2 ──→ Thread 2 (1 MB)
Client 3 ──→ Thread 3 (1 MB)
...
Client 1000 ──→ Thread 1000 (1 MB)

Context Switching도 빠름:
├─ Thread 1 → Thread 2: ~1 마이크로초
├─ Thread 2 → Thread 3: ~1 마이크로초
└─ 거의 무시할 수 있는 수준!
```

### **결론**
- `Thread` 기반 구조를 사용할 경우, 메모리 절감 + 속도 향상를 통해 더 많은 동시 사용자 처리가 가능하여 `MySQL` 은 `Thread` 기반 구조를 채택

## C. 스레드 기반의 문제

그러나 `Thread` 를 사용하기 때문에 `Process` 기반에서는 발생하지 않을 문제도 보유하게 됨.
- `메모리 공유` 로 인한 데이터 정합성 문제
- 하나의 스레드의 오류의 전파

### C-1: 메모리 공유로 인한 데이터 오염
- **상황**
	- 여러 `Thread` 가 같은 변수에 접근해서 값을 `1` 만큼 올리는 동작을 하는 것을 상상해보자.
```
Thread 1        Thread 2
  ↓              ↓
  └──→ [공유 메모리 변수] ←──┘
       count = 0

Thread 1에서: count = count + 1
Thread 2에서: count = count + 1
```

**문제 상황** -> `Race Condition`
```
기대값: count = 2
실제값: count = 1 (또는 다른 값!) ❌
```

**발생 원인**
- 실행 순서의 보장 X.
```
Time 1: Thread 1: count 읽음 (0)
Time 2: Thread 2: count 읽음 (0)  ← 아직 Thread 1의 변경이 반영 안 됨
Time 3: Thread 1: count + 1 = 1 쓰기
Time 4: Thread 2: count + 1 = 1 쓰기  ← 결과: 1
```

### C-2: 하나의 스레드 오류가 전체를 죽임
- 하나의 `thread` 오류가 서비스 전체에 영향을 끼칠 수 있음
```
모든 클라이언트 ──→ MySQL 서버 (하나의 프로세스) ──→ Thread 1, 2, 3...

Thread 2가 메모리 오류 (Segmentation Fault) 발생!
    ↓
프로세스 전체가 죽음 ❌
    ↓
모든 스레드가 중단 ❌
    ↓
모든 클라이언트 연결 끊김 😱
```

반면 `Process` 기반에서는 이러한 문제가 발생하지 않음.
```
프로세스 기반 (PostgreSQL):
Client 1 ──→ Process 1
Client 2 ──→ Process 2  ← 이 프로세스가 오류 발생!
Client 3 ──→ Process 3

Process 2가 죽어도:
├─ Process 1은 계속 실행 ✓
├─ Process 3도 계속 실행 ✓
└─ Client 2만 연결 끊김
```

### C-3. MySQL에서 Thread의 문제를 해결한 방법

####  해결책 1: MVCC (Multi-Version Concurrency Control)

**아이디어**: 데이터를 "버전 관리"처럼 처리

```
원본 데이터: name = "Alice", age = 25

Thread 1: UPDATE로 age를 30으로 변경하려고 함

MVCC가 하는 일:
├─ 원본 보존: Version 0 (age = 25) - 계속 유지
├─ 새 버전 생성: Version 1 (age = 30) - Thread 1이 수정 중
└─ Undo Log: 이전 버전 기록

동시에 다른 스레드들:
├─ Thread 2: SELECT name, age  → Version 0 읽음 (age = 25)
├─ Thread 3: SELECT name, age  → Version 0 읽음 (age = 25)
└─ Thread 1: UPDATE 완료        → Version 1 완성 (age = 30)

결과: 잠금 없이도 안전한 읽기 가능!
```

**효과**: Race Condition 방지, 동시성 증가

---

### 해결책 2: Fine-Grained Locking (세분화된 잠금)

**아이디어**: 전체 데이터베이스를 잠그지 말고, 필요한 부분만 잠금

```
❌ 과거 방식 (느림):
Buffer Pool 전체를 하나의 Lock으로 관리
    ↓
Thread 1이 Lock 획득 → 페이지 접근
Thread 2가 대기... → 다른 페이지에 접근하고 싶어도 못함
Thread 3도 대기...
결과: 많은 대기 시간

✅ 현대 MySQL:
Buffer Pool을 여러 Instance로 분할
├─ Instance 1 [Lock 1] → Thread 1이 사용
├─ Instance 2 [Lock 2] → Thread 2가 동시에 사용 ✓
├─ Instance 3 [Lock 3] → Thread 3도 동시에 사용 ✓
└─ Instance 4 [Lock 4] → Thread 4도 동시에 사용 ✓

결과: 대기 시간 극감!
```

**효과**: 스레드가 많아도 전체가 느려지지 않음

---

### 해결책 3: Signal Handling (신호 처리)

**아이디어**: 스레드 오류를 감지하고 격리

```
Thread 2에서 메모리 오류 발생!
    ↓
MySQL의 Signal Handler가 감지
    ↓
Thread 2가 가진 Lock/Mutex를 강제로 해제
    ↓
Thread 2만 종료 (다른 스레드는 계속 실행) ✓
    ↓
(하지만 완벽한 해결은 아님 - 메모리가 이미 오염되었을 수 있음)
```

**효과**: 일부 상황에서는 전체 시스템 붕괴 방지

## 7️⃣ PostgreSQL은 왜 프로세스를 고수했을까?

### 안정성의 철학

```
PostgreSQL의 생각:

"우리는 금융, 의료, 정부 시스템처럼 
데이터 손실이 절대 안 되는 분야도 사용할 것이다.

따라서 스레드의 복잡성보다 
프로세스의 강력한 격리가 더 중요하다."

각 클라이언트 ──→ 독립 프로세스 (물리적 격리)
    ↓
한 프로세스의 오류가 다른 프로세스에 영향 없음
    ↓
데이터 무결성 보장!
```

### PostgreSQL이 감수하는 비용

```
대신에...

1. 메모리 사용량이 많음
   └─ Client 1000명 = 5~20 GB 메모리 필요

2. Context Switching 오버헤드 많음
   └─ 1000개 프로세스 간 전환 = 시간 낭비

3. 하지만 안정성은 최고
   └─ 한 클라이언트의 오류가 다른 클라이언트에 영향 없음
```

## 8️⃣ MySQL이 PostgreSQL에게 지는 상황

### 시나리오 A: 복잡한 분석 쿼리 (OLAP)

```
쿼리: 100억 건의 데이터를 분석하고 집계

MySQL의 한계:
├─ 하나의 쿼리는 기본적으로 1개 스레드만 사용
├─ 32 코어 CPU가 있어도 1개 코어만 사용
└─ 처리 시간: 100초

PostgreSQL의 병렬 처리:
├─ 하나의 쿼리를 여러 프로세스로 분산
├─ 32 코어 CPU를 거의 모두 활용
└─ 처리 시간: 5초 (20배 빠름!)

결론: OLAP은 PostgreSQL이 훨씬 유리
```

---

### 시나리오 B: 장시간 실행 쿼리

```
상황: 읽기 쿼리 A가 1시간 이상 실행 중

MySQL (MVCC 문제):
├─ 쿼리 A가 오래된 버전을 참조하고 있음
├─ 그 동안 UPDATE가 1000만 건 발생
├─ Undo Log 크기: 점점 증가 (메모리 낭비)
├─ 디스크 I/O 폭증 (성능 저하)
└─ Purge Thread 과부하 (전체 시스템 느려짐)

PostgreSQL:
├─ VACUUM 작업이 명시적이고 예측 가능
├─ 필요할 때만 Lock을 잠시 획득하고 정리
└─ 성능이 더 예측 가능
```

## 9️⃣ 정리: 언제 뭘 써야 할까?

### MySQL 사용하기 좋은 상황

✅ **OLTP (짧은 쿼리 폭주)**

```
예: 웹 서비스 (SNS, 전자상거래)
- 각 쿼리 실행 시간: 100ms 미만
- 동시 접속자: 수천~수만
- 빠른 응답이 가장 중요

→ MySQL 최적
```

✅ **메모리가 제한적인 경우**

```
예: 클라우드 환경 (작은 VM)
- 사용 가능 메모리: 4GB
- 동시 사용자: 수백~수천

→ 프로세스 기반은 메모리 초과
→ MySQL의 스레드 기반이 필수
```

### PostgreSQL 사용하기 좋은 상황

✅ **OLAP (복잡한 분석)**

```
예: 데이터 웨어하우스
- 쿼리 실행 시간: 수분~수시간
- 병렬 처리가 중요
- 데이터 정합성 중요

→ PostgreSQL 최적
```

✅ **안정성이 생명인 경우**

```
예: 금융, 의료, 정부 시스템
- 데이터 손실 절대 불가
- 버그로 인한 시스템 다운 절대 불가
- 강력한 격리가 필수

→ PostgreSQL의 프로세스 기반 격리가 필수
```

✅ **데이터가 매우 큰 경우**

```
예: 테라바이트 단위 데이터 분석
- 메모리 사용량이 중요하지 않음
- 정확한 데이터 처리가 가장 중요

→ PostgreSQL
```

## 🔟 결론: 트레이드오프 (Trade-off)

```
MySQL은 우월한 것이 아니라 "최적화된 것"일 뿐입니다.

MySQL의 선택:
"성능을 극대화하되, 
 안정성의 마지노선을 소프트웨어(MVCC, Fine-grained Lock)로 관리한다"

PostgreSQL의 선택:
"성능은 조금 포기하되,
 OS의 프로세스 격리로 절대적인 안정성을 보장한다"

둘 다 합리적인 선택입니다. 
상황에 따라 최적의 도구가 다를 뿐입니다.
```

## 1️⃣1️⃣ 학부 시험 대비 핵심 정리

### Q1: MySQL이 스레드를 선택한 이유는?

**답**:

```
1. 메모리 효율: 스레드는 프로세스보다 메모리 사용 5배 적음
2. Context Switching: 스레드 전환이 프로세스 전환보다 1000배 빠름
3. 캐시 효율: 같은 주소 공간을 사용하므로 캐시 히트율 높음
4. 많은 동시 연결 처리 가능
```

### Q2: PostgreSQL이 프로세스를 고수한 이유는?

**답**:

```
1. 격리: 한 클라이언트 오류가 다른 클라이언트에 영향 없음
2. 안정성: OS 레벨에서 메모리 보호
3. 병렬 처리: 하나의 쿼리를 여러 프로세스로 분산 가능
4. 금융/의료 같은 안전성 중요 분야에 적합
```

### Q3: MySQL의 위험성은?

**답**:

```
한 스레드의 오류 → 전체 프로세스 죽음 → 모든 클라이언트 연결 끊김

해결책:
1. MVCC: 버전 관리로 Race Condition 방지
2. Fine-grained Locking: 세분화된 잠금으로 경합 감소
3. Signal Handling: 오류 감지 및 격리 시도
(하지만 완벽하지는 않음)
```

### Q4: Context Switching이 왜 느릴까?

**답**:

```
프로세스 전환:
- Page Table 변경 필요 (가상 주소 매핑이 완전히 다름)
- TLB (주소 변환 캐시) 초기화 필요
- CPU 캐시도 초기화됨
- 총 1~10ms 소요

스레드 전환:
- Page Table 변경 불필요 (같은 주소 공간)
- TLB 유지 가능
- CPU 캐시도 대부분 유효
- 총 1마이크로초 소요

→ 약 1000배의 차이!
```

## 1️⃣2️⃣ 추가 학습: 실무 관점

### MySQL을 쓸 때 조심할 점

```
1. 장시간 실행 쿼리 금지
   └─ Purge Lag 발생 → 전체 시스템 느려짐

2. 한 번에 너무 많은 UPDATE 금지
   └─ 메모리 폭증 → Undo Log 비대화

3. 복잡한 분석은 피하기
   └─ 병렬 처리 불가 → 시간 초과

4. 연결 풀 사용 필수
   └─ 스레드 생성 오버헤드 줄이기
```

### PostgreSQL을 쓸 때 장점

```
1. 복잡한 분석도 안심
   └─ 병렬 처리가 자동으로 최적화

2. 메모리 오염 걱정 없음
   └─ 프로세스가 물리적으로 격리됨

3. 데이터 무결성 최고
   └─ 많은 금융권에서 선택
```
